# Go style rules (auto-loaded for Go code)

## Форматирование и структура
- gofmt / goimports обязательно. Без неотформатированного кода в коммитах.
- Один бинарник с горутинами (не микросервисы). Точка входа: cmd/procluster.
- Внутренняя логика — в internal/ по пакетам зоны ответственности
  (ingest, aggregator, store, api, auth, indicators, admin, security).
- Доступ к БД — через интерфейсы repository, без прямых SQL по всему коду
  (нужно для будущей миграции SQLite → PostgreSQL).

## Ошибки
- Ошибки оборачивать с контекстом: fmt.Errorf("...: %w", err).
- Не игнорировать err (никаких _ = f() для важных вызовов).
- Никаких panic в рантайме обработки запросов/трейдов. panic только при старте
  на неустранимой ошибке конфигурации.

## Конкурентность
- Останов воркеров и горутин — через context.Context, без утечек горутин.
- Общение воркеров — через каналы / Redis, а не общие мутабельные структуры без защиты.
- Защищать общие данные sync.Mutex/RWMutex или каналами; гонок быть не должно.
- Запускать тесты с -race на конкурентном коде.

## Производительность (hot-path: ingest, aggregator, ws-hub)
- В обработке трейдов/рассылке WS избегать лишних аллокаций; переиспользовать буферы.
- Не делать блокирующий I/O (запись в ClickHouse) в горячем цикле приёма трейдов —
  батчить и писать асинхронно.

## Безопасность и ввод
- Весь внешний ввод (HTTP, WS-сообщения) валидируется до использования.
- Секретов в коде нет — только через env/config. Логи без паролей/токенов.
- (Для сетевого/auth/admin кода также применяется .claude/rules/security.md.)

## Тесты
- Юнит-тесты на критичную логику: округление, сжатие, агрегация по ТФ,
  формулы индикаторов. Примеры брать из docs/DATA_MODEL.md и docs/INDICATORS.md.
- Имена тестов описательные: TestTruncateVolume_DropsBelowTenth и т.п.

## Зависимости
- Минимум внешних зависимостей. Перед добавлением крупной библиотеки — обоснование
  в DECISIONS.md. Проверять go list -m all на уязвимости (см. фаза 15).
