## Легенда
- Модель: Opus = сложная логика/архитектура/безопасность; Sonnet = рутина/верстка/CRUD; Haiku = тривиальные subagent-задачи.
- SubAgent: рекомендуется делегировать (изоляция контекста / параллелизм).
- 🔒 = точка проверки безопасности в этой фазе.

## Точки безопасности (когда что проверять)
- Фаза 2/3: параметризация запросов к ClickHouse, валидация трейдов, отсутствие секретов.
- Фаза 4: лимиты пагинации, размер запроса, лимиты WS-соединений.
- Фаза 4: + атомарный лимит активных графических сессий по userId (Redis), last-wins, heartbeat.
- Фаза 8: корректность depth-sync (иначе данные искажаются), доступ к БД только внутри docker-сети.
- Фаза 9: argon2id, rate-limit login/register, lockout, httpOnly cookie, authZ.
- Фаза 12: проверка роли Admin на каждом эндпоинте, audit log, фоновая загрузка не блокирует воркеры.
- Фаза 14: SSH (только ключи, fail2ban), firewall, requirepass Redis, security-заголовки, TLS.
- Фаза 15: полный аудит по чек-листу docs/SECURITY.md с делегированием subagent'ам.

## Зависимости между фазами
- 5 требует 4 (данные с API). 6 требует 5. 8 требует 3 (depth stream рядом с ingest).
- 11 требует 6 (рендер) + 9 (права по тарифам). 12 требует 9 (роль Admin).
- 14 можно начинать каркасом параллельно после 0, добивать в конце.

## Правило выбора модели (коротко)
Если задача = "придумать как / посчитать формулу / держать 60fps / не сломать безопасность" → Opus.
Если задача = "сверстать по макету / написать CRUD / конфиг / тест по образцу / мелкая правка" → Sonnet.
Если задача внутри subagent = "найти и вернуть конспект / прогнать линтер" → можно Haiku.
