# ARCHITECTURE.md — PROCLUSTER

## Цель документа
Общая карта системы: какие сервисы есть, как данные текут от Binance до экрана.
Детали по каждой части — в отдельных доках (DATA_MODEL, CHART_ENGINE, DOM_SPEC и т.д.).

## Высокоуровневая схема потоков данных

Binance Futures/Spot WS (trades stream)
        │
        ▼
[ingest]  ── нормализация трейда (price, qty, isBuyerMaker, tradeId, time)
        │     сортировка по tradeId, защита от потери пакетов, reconnect
        ▼
[aggregator] ── черновая агрегация в Redis (кластер текущей свечи на лету)
        │        при закрытии интервала ТФ → запись в ClickHouse
        ▼
   ClickHouse (clusters_futures / clusters_spot, по ТФ и уровням сжатия)
        │
        ▼
[api] REST (история свечей, N последних) + WS hub (live кластер каждые 100-500ms)
        │
        ▼
Frontend (React) → chart2d (Canvas2D) + DOM + indicators   [PixiJS/WebGL отложен, см. DECISIONS.md]


Параллельно:
- data.binance.vision loader: ручной запуск из админки/CLI, грузит тики
  за диапазон дат, агрегирует и пишет в ClickHouse с прогресс-баром.
- DOM snapshot worker: снапшот стакана раз в 1м (futures) / 15м (spot),
  агрегирует по шагу сжатия тикера, пишет в clusters_*_dom.
- Fear & Greed worker: тянет индекс раз в день, кэширует.

## Сервисы внутри одного Go-бинарника (горутины, не микросервисы)
- ingest-worker      — подключения к Binance WS, по одному на символ+рынок
- aggregator         — потоковая агрегация трейдов
- history-loader     — загрузка/агрегация исторических тиков (по команде)
- dom-snapshotter    — снапшоты стакана
- fng-fetcher        — Fear & Greed
- http-server        — REST + статика API
- ws-hub             — рассылка live-данных подписчикам по символам/ТФ
- scheduler          — крон-подобные задачи (TTL, обслуживание)

Все воркеры общаются через каналы Go и Redis. Останов — через context.Context.

## Хранилища и зоны ответственности
- ClickHouse: market data (кластеры, футпринт, снапшоты стакана). Большие объёмы, аналитика, TTL.
- Redis: горячая черновая агрегация текущих свечей, кэш последних N свечей, кэш пресетов индикаторов, rate-limit счётчики.
- SQLite: пользователи, сессии, настройки пользователей, тарифы, платежи (метаданные), пресеты.

## Кэш-стратегия
- При открытии графика фронт грузит из кэша ТОЛЬКО последние 700 свечей по ТФ.
- Скролл в прошлое → подгрузка из ClickHouse по запросу (пагинация по времени).
- Live-обновление текущей свечи приходит по WS.

## Лимиты истории по типам пользователей (детали в AUTH_PLANS.md)
- Гость: 1 неделя
- Free/авторизован: 6 месяцев
- Pro/VIP: вся доступная история из БД

## Сетевая топология деплоя
- Caddy (reverse proxy, авто-TLS) →
    - procluster.online → лендинг (статика/SSR)
    - chart.procluster.online → React-приложение + /api проксируется в Go
- Go-бинарник, ClickHouse, Redis — контейнеры в docker-compose, наружу торчит только Caddy.
- ClickHouse/Redis/SQLite НЕ доступны из интернета (только внутренняя docker-сеть).

## Формат обмена
JSON везде на старте. Переход на MessagePack — только если профайлинг покажет bottleneck (записать в DECISIONS.md).

## Принципы, которые нельзя нарушать
1. Движок графика изолирован и не знает про конкретные UI-компоненты.
2. Любая агрегация/сжатие — единая реализация (один источник правды), используется и для live, и для истории, и для стакана.
3. Параметры сжатия тикеров берутся из конфигурации в БД/админке, НЕ хардкодятся.
