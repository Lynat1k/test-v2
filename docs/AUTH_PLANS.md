# AUTH_PLANS.md — PROCLUSTER

## Авторизация
- Регистрация/вход по email + пароль.
- Вход через Google OAuth.
- Сессии: JWT (access короткий + refresh) ИЛИ серверные сессии — выбрать, записать в DECISIONS.md.
  Рекомендация: refresh-токен в httpOnly cookie, access — в памяти фронта.
- Пароли: bcrypt/argon2id (argon2id предпочтительно). Никогда не хранить в открытом виде.
- Хранилище: SQLite на старте → план миграции на PostgreSQL (см. ниже).

## Типы пользователей
Free | Pro | VIP | Admin

## Лимиты истории графика
- Гость (не авторизован): 1 неделя.
- Free: 6 месяцев.
- Pro: вся доступная история из БД.
- VIP: вся история.
- Admin: всё + админка.

## Профиль пользователя
- Ник, аватар (из набора предложенных), email, дата регистрации.
- Поле для токена Telegram (для алертов; функционал у VIP/Admin).
- Смена пароля.
- Уровень подписки (Free/Pro/VIP/Admin).
- Дата оплаты, дата истечения, осталось дней, статус (активна/на паузе/закончена).
- В конце профиля — блок тарифов со сравнением и кнопкой "Активировать".

## Сохранение настроек пользователя
Хранить (SQLite для авторизованных, localStorage как fallback/для гостя):
последний ТФ, тикер, тип биржи (рынок), вид свечей, уровень сжатия,
рабочее пространство, тема сайта, язык, набор и настройки индикаторов на графике.

## Гейтинг по тарифу (правило)
Все лимиты проверяются НА БЭКЕНДЕ (источник правды), фронт только подсказывает UI.
Никогда не доверять клиенту в вопросе доступной истории/функций.

## План миграции SQLite → PostgreSQL
- Доступ к БД через интерфейс repository, без прямых SQL по всему коду.
- При росте RAM/нагрузки — переключение драйвера, миграция схемы. Записать триггер в DECISIONS.md.

## Ограничение одновременных активных графических сессий

Цель: ограничить число одновременно работающих окон графика на ОДИН аккаунт
по тарифу. Закрывает кейс «один аккаунт в разных вкладках/браузерах».
(Мультиаккаунты технически полностью не решаются — см. примечание ниже.)

### Что считаем "активной графической сессией"
- Активная сессия = открытое WS-соединение, подписанное на live-данные графика.
- Учёт ведётся на бэкенде по userId (источник правды), НЕ по доверию фронту.
- Гость (не авторизован) считается по анонимному идентификатору сессии/IP отдельно
  (мягкий лимит, не строгий).

### Лимиты по тарифу (числа берутся из конфигурации/админки, не хардкод)
- Free:  1 активная графическая сессия
- Pro:   2 (или по конфигурации; согласуется с "2 графика")
- VIP:   2 (или по конфигурации)
- Admin: без лимита

Примечание: лимит окон графика и лимит "графиков в рабочем пространстве" — РАЗНЫЕ вещи.
Здесь речь про одновременные ПОДКЛЮЧЕНИЯ (вкладки/браузеры), а не про layout 1/2 графика.

### Хранение в Redis
- Ключ-множество активных сессий пользователя:
  chart_sessions:{userId} → набор sessionId с временем последнего heartbeat.
  Реализация: Redis Sorted Set (ZADD), score = timestamp последнего heartbeat.
- TTL/протухание: сессия без heartbeat дольше N секунд считается мёртвой и удаляется
  (ZREMRANGEBYSCORE по порогу now - N). Рекомендация: heartbeat каждые 10с, порог 30с.
- Атомарность: проверку лимита и добавление сессии делать атомарно
  (Lua-скрипт или MULTI/EXEC), чтобы две одновременные вкладки не проскочили обе.

### Политика при превышении лимита (выбрана: last-wins)
- При новом WS-подключении к графику:
  1. Очистить протухшие сессии пользователя.
  2. Если число живых сессий < лимита → зарегистрировать новую, разрешить рендер.
  3. Если = лимиту → применить LAST-WINS:
     - новой сессии разрешаем стать активной,
     - самой старой активной сессии шлём по WS управляющее сообщение
       {type:"session_evicted", reason:"limit"} → фронт переводит её в пассив
       (стоп рендер, отписка от данных, показать плашку "График открыт в другом окне").
- Альтернатива (reject) допустима для отдельных тарифов через конфиг:
  новой сессии отказать с {type:"session_rejected"} и подсказкой про тариф.
  Выбор политики хранить в конфигурации тарифа.

### Управляющие сообщения WS (контракт)
- Сервер → клиент:
  {type:"session_active"}    — эта сессия активна, рендер разрешён
  {type:"session_evicted"}   — вытеснена более новой, уйти в пассив
  {type:"session_rejected"}  — отказ по лимиту тарифа
- Клиент → сервер:
  {type:"chart_subscribe", symbol, market, timeframe}
  {type:"heartbeat"}         — каждые ~10с, обновляет score сессии
  {type:"chart_unsubscribe"} / закрытие сокета — освобождает слот

### Взаимодействие с клиентским leader-election
- Внутри одного браузера вкладки сами выбирают лидера (BroadcastChannel) — это
  снижает число подключений до одного на браузер ещё ДО сервера.
- Серверный лимит — второй рубеж, закрывает РАЗНЫЕ браузеры/устройства одного аккаунта.

### Мультиаккаунты (примечание, не реализуем строго на старте)
Полностью не решается. На старте: обязательное подтверждение email + мягкий
rate-limit регистраций по IP. Основная защита — в ценности платных тарифов
(история, пресеты, алерты, синхронизация), а не в технических барьерах.
