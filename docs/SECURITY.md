## Секреты
- НИКОГДА не коммитить секреты (токены, пароли, ключи Google OAuth, креды БД).
- Только .env (в .gitignore) + .env.example с пустыми значениями.
- GitHub-токены для Claude Code передавать через переменные окружения/gh CLI,
  не вписывать в промты и не коммитить.
- Если секрет попал в репозиторий/чат — считать скомпрометированным, отозвать и заменить.

## Ввод и API
- Валидировать ВЕСЬ пользовательский ввод на бэкенде (типы, диапазоны, длины).
- Параметризованные запросы везде (ClickHouse/SQLite) — без конкатенации SQL.
- Лимиты на размер тела запроса, на параметры пагинации (нельзя запросить "всё").
- CORS: только нужные origin (procluster.online, chart.procluster.online).
- Заголовки безопасности: CSP, HSTS, X-Content-Type-Options, X-Frame-Options.

## Аутентификация
- Argon2id для паролей. Соль на пользователя.
- Защита от перебора логина: rate-limit на /login, прогрессивная задержка, lockout после N попыток.
- Refresh-токен в httpOnly+Secure+SameSite cookie. Ротация refresh.
- Google OAuth: проверка state, валидация id_token.
- Проверка роли (Admin) на каждом админ-эндпоинте.

## Rate limiting / защита от DDoS и спама запросами
- Per-IP и per-user rate limit (Redis-счётчики, token bucket / sliding window).
- Отдельные, более жёсткие лимиты на: /login, /register, восстановление пароля, админ-загрузку.
- WS: лимит подписок на соединение, лимит соединений на IP, heartbeat/timeout.
- Защита тяжёлых эндпоинтов (история): обязательная пагинация, кэш, ограничение глубины для тарифа.
- На уровне Caddy/Nginx: rate limit, лимит размера запроса, таймауты.
- Cloudflare (или аналог) перед доменом — рекомендация против объёмного L7 DDoS.

## Лимит активных графических сессий (anti-abuse)
- Лимит одновременных графических WS-сессий на userId по тарифу — на бэкенде (Redis).
- Проверка лимита и регистрация сессии — АТОМАРНО (Lua/MULTI), без гонок.
- Heartbeat + протухание мёртвых сессий, чтобы лимит не "залипал" после обрыва связи.
- Клиенту НЕ доверяем: вытеснение/отказ решает сервер, фронт только отражает.
- Регистрация: обязательное подтверждение email; мягкий rate-limit регистраций по IP.


## Защита сервера (VPS)
- SSH: запретить вход по паролю (только ключи), сменить порт по желанию,
  disable root login, fail2ban на ssh (бан после N неудачных попыток).
- Firewall (ufw/iptables): открыты только 80/443 и SSH (желательно по whitelist IP).
- Базы (ClickHouse/Redis/SQLite) НЕ доступны из интернета — только внутренняя docker-сеть.
- Redis: requirepass + bind на внутренний интерфейс. ClickHouse: пользователь с паролем, не default.
- Регулярные обновления ОС, автоматические security-патчи.
- Бэкапы: ClickHouse (history дорого восстанавливать) + SQLite (пользователи!).



## Чек-лист аудита кода (использовать на фазах безопасности)
- [ ] Нет секретов в репозитории (история git тоже).
- [ ] Все вводы валидируются, SQL параметризован.
- [ ] AuthN/AuthZ на всех приватных и админ-маршрутах.
- [ ] Rate-limit на login/register/recovery/admin.
- [ ] Лимиты пагинации и размеров запросов.
- [ ] CORS/CSP/security-заголовки настроены.
- [ ] Зависимости без известных уязвимостей (go list -m, npm audit).
- [ ] WS: лимиты соединений/подписок, валидация сообщений.
- [ ] Логи без чувствительных данных (пароли, токены).
- [ ] Ошибки наружу без stack trace/внутренних деталей.
- [ ] Лимит графических сессий: считается на бэкенде по userId, клиенту не доверяем.
- [ ] Проверка+регистрация сессии атомарна (Lua/MULTI), гонка не превышает лимит.
- [ ] Протухшие сессии чистятся по heartbeat; слот освобождается после обрыва связи.
- [ ] last-wins: вытесненная сессия реально уходит в пассив, обе одновременно не активны.
- [ ] Нельзя получать данные без heartbeat в обход лимита.
- [ ] Гость считается отдельно от авторизованных пользователей.
