# DATA_MODEL.md — PROCLUSTER

## Цель
Точные правила хранения и агрегации рыночных данных. Это критичный документ —
от него зависит точность футпринта (цель: близко к ATAS / Tiger Trade).

## Источник трейдов
- **Spot**: Binance `@trade` stream (индивидуальные трейды, поле `t` = tradeId).
- **Futures (USDT-M)**: Binance `@aggTrade` stream (агрегированные за ~100ms, поле `a` = aggregate trade ID).
  На фьючерсах individual `@trade` stream отсутствует — это ограничение Binance, а не наш выбор.
  aggTrade сохраняет цену, сторону (isBuyerMaker) и объём — достаточно для футпринта.
- Оба формата маппятся в единый `model.Trade`.

### Правила интерпретации
- isBuyerMaker == true  → агрессор ПРОДАВЕЦ → это ASK-объём (продажа в бид, красный).
- isBuyerMaker == false → агрессор ПОКУПАТЕЛЬ → BUY/BID-сторона (зелёный).
- Сортировать строго по tradeId по возрастанию.
- Не терять пакеты: при реконнекте дозапросить пропущенные tradeId через REST, проверить непрерывность.
- Цена агрегации — фактическая цена трейда, привязка к ценовому ряду по шагу сжатия.
- Время — trade time (не время получения).

## Понятие "тик" и "сжатие"
- Тик = минимальный шаг цены инструмента (priceTick).
  - BTCUSDT Futures: 0.1 $
  - BTCUSDT Spot: 0.01 $
- "Сжатие N" = объединение N тиков в один ценовой ряд кластера.
  - Futures сжатие 25 → ряд = 25 * 0.1 = 2.5 $
  - Spot сжатие 500 → ряд = 500 * 0.01 = 5.0 $
- Базовое (минимальное) сжатие на символ задаётся в админке и хранится в БД.
- Пользователь может выбрать кратные уровни: base, base*2, base*3, ... максимум 10 уровней.
  Пример futures base=25: 25, 50, 75, 100, ... (объединение на лету или из кэша).

## Округление объёмов (ОБЯЗАТЕЛЬНО, единое правило)
Объём в монетах округляется до 1 знака после запятой (TRUNCATE — отбрасывание без округления):
- 5.1256 → 5.1
- 5.627  → 5.6
- 0.0125 → 0.0  (пишем 0)
- 0.85   → 0.8
Правило: TRUNCATE до 1 знака (math.Trunc(v*10)/10). ЗАФИКСИРОВАНО в DECISIONS.md.

## Модель ценового ряда (кластера)
Для каждой свечи (символ, рынок, ТФ, время открытия) хранится набор ценовых рядов:
- price_level (нижняя граница ряда по шагу сжатия)
- bid_volume (агрессивные покупки, BUY)
- ask_volume (агрессивные продажи, SELL)
- delta = bid_volume - ask_volume (можно вычислять, но удобнее хранить для скорости)
- total_volume = bid_volume + ask_volume
Плюс агрегаты свечи: open, high, low, close, total_bid, total_ask, total_delta, total_volume, trades_count.

## OHLC свечей (Open/Close)
**Правило:** Open = цена ПЕРВОГО по времени трейда минуты, Close = цена ПОСЛЕДНЕГО.
- High = max(price_level), Low = min(price_level) — как и раньше.
- Open/Close хранятся как колонки `open_price`/`close_price` в таблицах clusters_*.
- Все строки кластера одной свечи имеют одинаковые open_price/close_price.
- Rollup старших ТФ: open = open первой 1m свечи интервала, close = close последней.
- Legacy-свечи (open_price=0): fallback open=low, close=high.
- Для определения first/last trade: primary key = trade time, tie-breaker = tradeId.
- Используется ОРИГИНАЛЬНАЯ цена трейда (trade.Price), НЕ синтетическая цена бакета.

## Имбаланс (для раскраски и индикаторов)
- Диагональный имбаланс (как ATAS): сравнение ask[price] vs bid[price - 1 ряд].
- Раскраска цифр: имбаланс > 300% → ask зелёным / bid красным соответственно
  [сверить направление цвета при реализации с CHART_ENGINE.md].

## Хранение: ClickHouse
Две раздельные таблицы для кластеров (свечи/футпринт):
- clusters_futures
- clusters_spot
у каждой собственный TTL.

Хранение кластеров — ОТДЕЛЬНЫЕ СТРОКИ на ценовой ряд (не вложенные массивы),
для гибких выборок и агрегации.

Примерная структура (адаптировать под ClickHouse при реализации):
clusters_futures (
  symbol         LowCardinality(String),
  timeframe      LowCardinality(String),
  candle_open    DateTime64(3),
  price_level    Decimal(...),       -- нижняя граница ряда
  bid_volume     Decimal(.,1),
  ask_volume     Decimal(.,1),
  compression    UInt16,             -- базовый уровень сжатия, на котором записано
  open_price     Decimal(18,2),      -- цена первого трейда минуты (0 = legacy)
  close_price    Decimal(18,2)       -- цена последнего трейда минуты (0 = legacy)
) ENGINE = MergeTree
PARTITION BY toYYYYMM(candle_open)
ORDER BY (symbol, timeframe, candle_open, price_level)
TTL candle_open + INTERVAL 1 YEAR    -- BTCUSDT futures: 1 год

clusters_spot: то же, TTL 3 года, compression base 500.

## Агрегация по таймфреймам
- Сырьё агрегируем согласно ТФ: для 1m — раз в минуту, для 1h — раз в час и т.д.
- Старшие ТФ можно строить из младших (rollup) либо из сырья — выбрать один путь,
  записать в DECISIONS.md. Рекомендация: хранить минимальный ТФ детально,
  старшие — materialized views / rollup, чтобы не дублировать объём.
- Объединение уровней сжатия (base*k) — на лету из base или из Redis-кэша.

## Кэш последних свечей
- Redis: по ключу (symbol, market, timeframe) хранить последние 700 свечей готовыми к отдаче.
- Текущая (незакрытая) свеча — отдельно в Redis, обновляется на каждом трейде.

## Историческая загрузка (data.binance.vision)
- Запуск вручную из админки/CLI: символ + рынок + диапазон дат.
- Скачивает архивы тиков, парсит, прогоняет через ТУ ЖЕ агрегацию, что и live.
- Прогресс-бар: этапы download → parse → aggregate → insert, % и ETA.
- Идемпотентность: повторная загрузка того же диапазона не должна дублировать данные
  (ReplacingMergeTree или предварительная очистка диапазона).
